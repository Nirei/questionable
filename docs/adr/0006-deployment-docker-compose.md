# 6. Deployment: Docker image built on-host, no CI write access

Date: 2026-05-23

## Status

Accepted.

## Context

We need a deployment pipeline that gets small, iterative changes in front
of real users (and a real federation peer) without accumulating risk for
the final push. Two competing goals:

1. **Move fast.** Every merge to `main` should be deployable; the
   review-to-live latency should be minutes.
2. **Don't expand the trust surface.** This project is built by an LLM
   agent. Even with conscientious code review, a malicious dependency, a
   prompt injection during docs browsing, or a catastrophic logic error
   could land in `main`. Whatever pipeline we build, it must not give the
   agent — or anything controlled by the agent — write access to a host
   that runs the operator's other services.

The natural cloud answer ("CI rsyncs an artefact over SSH and restarts
systemd") collapses option 2: a poisoned commit lands on the host the
moment it lands on `main`, with the host's full local filesystem and
network reach. Free PaaS hosts (Fly free tier, Render free, Heroku) are
either retired or impose constraints incompatible with long-lived
federation (ephemeral databases, rotating URLs).

## Decision

- **Artifact is a Docker image** built from a multi-stage `Dockerfile`
  in this repo. Builder stage runs `mix release` against
  `hexpm/elixir:1.18.4-erlang-27.3.4-debian-bookworm-...`; runner stage
  is `debian:bookworm-...-slim` with only the runtime libraries the
  release needs. Image runs as non-root, with `tini` as PID 1 for clean
  signal handling.
- **No container registry.** The image is built on the operator's VPS
  from the repo source. Loop:

  ```bash
  git pull
  docker compose up -d --build
  ```

  This trades a bit of build time on the host for zero registry
  complexity and zero credentials anywhere.
- **No CI write access to the host.** GitHub Actions runs `mix check`
  on every PR. It does not have, and will never be given, an SSH key,
  a registry push token, or any other credential that can reach the
  production VPS. The operator pulls and restarts manually, by choice.
- **`compose.yaml` ships in the repo** and defines the full stack: the
  app service + `postgres:16-alpine` with a named volume for
  persistence. The app binds to `127.0.0.1` only; the operator's
  existing nginx terminates TLS and reverse-proxies.
- **Env via `.env`** (gitignored), with `.env.example` checked in as
  the canonical template. Secrets (`SECRET_KEY_BASE`, `PG_PASSWORD`,
  `RELEASE_COOKIE`) are generated by the operator and never leave
  the VPS.
- **Persistence is a named volume** (`pgdata`). Backups are the
  operator's responsibility — but losing the volume loses the
  ActivityPub actor private keys, which permanently breaks federation
  (other servers cache our `publicKey` and have no rotation
  protocol). Backup discipline is therefore part of operating this
  project, documented in the README.

## Consequences

- **Trust surface stays small.** A compromised dependency reaches the
  operator's host only after (a) the malicious diff appears in a PR
  the operator reviews, (b) the operator merges, (c) the operator
  pulls, (d) the operator restarts. Each step is a review checkpoint
  the operator controls.
- **No CI/CD pipeline to maintain.** No registry account, no deploy
  keys, no rotation. The CI in this repo only validates.
- **Slower than auto-deploy.** Iteration latency includes the
  operator's "I want to update now" decision and a ~2-minute build on
  the VPS. For a side project this is the right trade.
- **Container isolation, not host isolation.** A successful RCE in
  the Phoenix app still runs inside a container with no host volume
  mounts beyond `pgdata`. It cannot read the operator's other
  services' files or call into other systemd units. Reaching the host
  requires both an app-level RCE and a container escape.
- **The `Dockerfile` is the deploy contract.** Anything the app needs
  at runtime must be installable in the runner stage's apt layer. If
  Phase 1+ adds dependencies on system libraries (e.g. for image
  processing), they go there and the image grows accordingly.
- **No staging environment.** `main` is prod. Pre-merge checks (CI,
  the headless-screenshot rig planned for Phase 5, and manual
  review) are where confidence is built. Federation peers see only
  what's been merged.
